# ProofForge Specification v0.2

Status: Draft

Date: 2026-03-28

## 1. Назначение

ProofForge — легковесная система для описания, исполнения и валидации тестов SAP бизнес-процессов.

Система должна закрывать полный цикл:

1. формализация сценария тестирования;
2. копирование и адаптация сценария под конкретный бизнес-кейс;
3. сборка тестового плана под фазу проекта;
4. запуск и сопровождение прогонов;
5. фиксация валидаций и дефектов;
6. просмотр прогресса и результатов.

Проект реализуется в подходе Specification Driven Development: сначала фиксируются сущности, процессы, статусы и ограничения, после этого допускается реализация.

## 2. Цели Продукта

Основные цели:

- использовать единую сущность `Test Scenario` и для шаблонов, и для конкретных кейсов, избегая лишнего уровня абстракции;
- поддержать ручные и роботизированные шаги в одной модели;
- поддержать несколько валидаций для одного шага;
- сделать прозрачной трассировку от тестового плана до дефекта;
- сохранить решение простым в эксплуатации: минималистичный UI, хранение в JSON, развертывание на одном VPS в Docker.

## 3. Границы MVP

В MVP система должна:

- позволять создавать и редактировать `Test Scenario`, `Test Plan`, `Run`, `Defect`;
- позволять копировать сценарий для создания конкретного кейса с другими параметрами;
- достаточно редактирование в виде текста JSON с валидацией схемы;
- поддерживать последовательное исполнение шагов и фиксацию результата каждого шага;
- результатами шагов могут быть созданные сущности в системе SAP — нужно фиксировать их номера, содержание, а также валидацию этих шагов;
- поддерживать отдельные валидации по результатам шага;
- позволять связывать дефект с конкретным прогоном и шагом;
- показывать прогресс исполнения плана и статусы прогонов;
- хранить данные в файлах JSON или совместимом простом формате;
- поддерживать вложения: скриншоты и JSON-снимки SAP-документов.

Продукт должен поддерживать два режима работы:

- `manager_mode` — обычный веб-интерфейс без связи с SAP; используется для планирования тестов, просмотра прогонов, дефектов и результатов;
- `consultant_mode` — тот же интерфейс, запущенный внутри Electron-приложения; этот режим может вызывать локальный интеграционный слой для чтения данных из SAP.

Ограничение MVP: `consultant_mode` (Electron + SAP-коннектор) не входит в MVP. В MVP артефакты загружаются вручную (скриншоты, JSON-файлы). SAP-коннектор реализуется как первая итерация после MVP.

Приоритет интеграций после базового ядра:

1. SAP (Electron-коннектор);
2. почтовые уведомления;
3. Jira.

В MVP система не обязана:

- заменять полноценный issue tracker уровня Jira;
- содержать сложный workflow engine;
- использовать тяжелую СУБД;
- поддерживать высокую нагрузку или сложную многокомандную организационную структуру;
- поддерживать multi-tenant режим внутри одного развертывания.

## 4. Технологический Стек

- **Backend:** Node.js
- **Frontend:** React
- **Хранение данных:** JSON-файлы
- **Развертывание:** Docker на VPS
- **Язык UI:** English (MVP)
- **Electron:** post-MVP (SAP-коннектор)

## 5. Пользовательские Роли

Одна учетная запись может совмещать несколько ролей.

- `Administrator` — управляет настройками, пользователями и системными справочниками.
- `Test Designer` — проектирует сценарии.
- `Test Lead` — формирует тестовые планы и контролирует прогресс.
- `Executor` — исполняет шаги теста.
- `Validator` — подтверждает корректность результатов и артефактов.
- `Observer` — просматривает статусы, отчёты и дефекты без изменения данных.

Робот рассматривается не как пользовательская роль, а как тип исполнителя шага.

В MVP роли являются тегами на пользователе и не ограничивают действий. Разграничение прав — post-MVP.

### 5.1 Аутентификация

Система разворачивается на публичном IP, поэтому аутентификация обязательна.

В MVP используются индивидуальные локальные учётные записи пользователей.

Минимальная модель учётной записи:

- уникальный `id`;
- уникальный `email`;
- `password_hash`;
- отображаемое имя;
- набор ролей или тегов пользователя;
- статус: `active` / `disabled`.

Пользователь входит в систему по `email + password`.

Пароли в открытом виде не хранятся. В данных и резервных копиях хранится только `password_hash`.

## 6. Предметная Модель

Общие правила модели данных:

- каждая сущность имеет обязательные системные поля: `id`, `entity_type`, `created_at`, `created_by`, `updated_at`, `updated_by`, `tags`, `custom_fields`;
- `id` — числовой автоинкремент;
- поля `created_by` и `updated_by` содержат `user_id` автора изменения;
- теги (например `OTC`, `R2R`, `P2P`, `E2E`) используются для классификации по бизнес-процессам, фильтрации и группировки; они не являются частью ID;
- обязательные и опциональные бизнес-поля определяются типом сущности;
- произвольные пользовательские поля допускаются только внутри объекта `custom_fields`; произвольные top-level поля не допускаются;
- `tags` хранятся как массив уникальных строк;
- все межсущностные ссылки хранятся по стабильным `id`.

### 6.1 Test Step

`Test Step` — атомарная единица тестирования. Хранится как часть `Test Scenario`.

Свойства шага:

- уникальный идентификатор (в пределах сценария);
- порядковый номер внутри сценария;
- краткое название;
- описание действия;
- тип исполнителя: `human` или `robot`;
- роль исполнителя;
- тип действия: например `transaction`, `document`, `report`, `other`;
- входные параметры;
- предусловия;
- ожидаемый результат;
- набор шаблонов валидации;
- признак обязательности шага;
- признак возможности продолжения после ошибки.

Шаг должен поддерживать два независимых слоя:

- `execution` — факт выполнения шага;
- `validation` — подтверждение правильности результата шага.

Ограничение MVP: сценарий представляет собой линейную последовательность шагов. Ветвления и параллельные ветки не поддерживаются.

### 6.2 Step Validation

`Step Validation` — отдельная проверка результата одного шага.

Один шаг может иметь:

- ноль валидаций;
- одну валидацию;
- несколько независимых валидаций.

Примеры:

- проверка созданного документа;
- проверка проводок;
- проверка отражения в отчётности;
- подтверждение другим участником процесса.

Свойства валидации:

- идентификатор;
- ссылка на `Step Execution`;
- ссылка на исходный шаблон валидации при наличии;
- название проверки;
- описание критерия;
- ответственный валидатор;
- статус;
- комментарий;
- ссылки на артефакты или доказательства.

На уровне `Test Step` хранится шаблон валидации. Во время `Run` на его основе создаётся фактическая `Step Validation`, связанная с конкретным `Step Execution`.

### 6.3 Test Scenario

`Test Scenario` — единственная сущность для описания тестов. Используется и как шаблон, и как конкретный кейс.

Разделение "шаблон vs конкретный кейс" — вопрос конвенции (именования, тегов), а не типа сущности. Любой сценарий можно копировать, любой можно запустить.

Свойства сценария:

- идентификатор;
- код или краткое имя;
- бизнес-цель;
- описание;
- упорядоченный список `Test Step`;
- бизнес-контекст: компания, завод, тип документа, контрагент и т.д.;
- ожидаемый бизнес-результат;
- `parent_id` — ссылка на сценарий-источник при копировании (опционально);
- приоритет;
- теги;
- версия;
- статус.

Операция "создать кейс" = скопировать сценарий → подставить конкретные параметры → при необходимости изменить шаги. `parent_id` сохраняет трассировку к оригиналу.

### 6.4 Test Plan

`Test Plan` — набор `Test Scenario`, который исполняется в рамках проектной фазы.

Связь между `Test Plan` и `Test Scenario` — many-to-many: один сценарий может входить в несколько планов (например, в Functional Test и в Integration Test).

Единица учёта прогресса плана — `Test Scenario`, а не шаг и не отдельная попытка выполнения.

Сценарий считается исполненным в рамках плана, если по нему существует хотя бы один `Run`, связанный с этим `Test Plan`, и этот `Run` достиг терминального статуса, отличного от `cancelled`.

Прогресс `Test Plan` рассчитывается по формуле:

`executed_scenarios / total_scenarios * 100`

Свойства плана:

- идентификатор;
- название;
- фаза: `functional`, `integration`, `uat`;
- период исполнения;
- владелец;
- список входящих `Test Scenario` с назначениями и приоритетами;
- общий статус.

### 6.5 Run

`Run` — конкретный прогон теста на основе `Test Scenario`.

`Run` создаётся из `Test Scenario`. При создании система делает снимок (snapshot) сценария и его шагов, чтобы последующие изменения сценария не повлияли на уже запущенный прогон.

Свойства прогона:

- идентификатор;
- ссылка на `Test Plan` при наличии;
- ссылка на исходный `Test Scenario`;
- снимок сценария на момент старта;
- исполнитель или набор исполнителей;
- плановая дата;
- фактическое начало и завершение;
- статус;
- список фактических `Step Execution`;
- итоговый результат;
- комментарий по прогону.

Терминальные статусы `Run` для расчёта прогресса плана:

- `completed`;
- `blocked`.

### 6.6 Step Execution

`Step Execution` — фактический результат выполнения шага в рамках конкретного `Run`.

Один `Test Step` в рамках одного `Run` может иметь несколько `Step Execution` (при перепрогоне). Текущий статус шага определяется последним `Step Execution`.

Свойства:

- идентификатор;
- ссылка на `Run`;
- ссылка на исходный `Test Step`;
- номер попытки;
- исполнитель;
- время начала и завершения;
- статус исполнения;
- фактические значения параметров;
- комментарий;
- артефакты (скриншоты, файлы);
- список зафиксированных SAP-объектов;
- список фактических `Step Validation`.

Каждый зафиксированный SAP-объект должен, как минимум, хранить:

- `source_system`;
- `object_type`;
- `object_id` или `object_number`;
- `payload_json` или `payload_text`;
- `captured_at`.

### 6.7 Defect

`Defect` — выявленная проблема, найденная в ходе исполнения или валидации.

Свойства дефекта:

- идентификатор;
- краткое название;
- описание;
- источник: `Run`, `Step Execution`, `Step Validation`;
- severity;
- priority;
- статус;
- автор;
- ответственный;
- ссылки на артефакты;
- шаги воспроизведения;
- связь с `Test Plan`, `Test Scenario`, `Run`, `Test Step`.

## 7. Связи Между Сущностями

- один `Test Scenario` содержит от одного до многих `Test Step`;
- один `Test Scenario` может быть скопирован в другой (`parent_id`);
- один `Test Plan` содержит много `Test Scenario` (many-to-many);
- один `Test Scenario` может входить в несколько `Test Plan`;
- один `Test Scenario` может иметь много `Run`;
- один `Run` содержит много `Step Execution`;
- один `Test Step` в одном `Run` может иметь несколько `Step Execution` (перепрогон);
- один `Step Execution` может иметь много `Step Validation`;
- один `Defect` должен быть связан минимум с одним объектом исполнения.

- `ready`

### 8.2 Статусы Test Plan

- `draft`

### 8.3 Статусы Run

- `draft`

- `in_progress`
- `completed`

- `passed_with_comments`

- `blocked`

### 8.5 Статусы Step Validation

- `pending`

### 8.6 Статусы Defect

- `open`
- `in_progress`
- `fixed`
- `ready_for_retest`
- `closed`
- `rejected`

## 9. Основные Пользовательские Сценарии

### 9.1 Проектирование

1. Пользователь создаёт `Test Scenario`.
2. В сценарии описываются шаги, параметры и ожидаемые результаты.
3. Для шагов задаются шаблоны валидации.

### 9.2 Подготовка конкретного кейса

1. Пользователь копирует существующий сценарий.
2. В копии подставляются конкретные параметры (компания, документ, контрагент).
3. При необходимости корректируются шаги.
4. Сценарий переводится в состояние `ready`.

### 9.3 Планирование

1. Пользователь формирует `Test Plan`.
2. В план добавляются конкретные `Test Scenario`.
3. По сценариям назначаются исполнители, даты и приоритет.
4. Прогресс плана в дальнейшем считается по количеству сценариев, для которых появился исполненный `Run`.

### 9.4 Исполнение

1. Пользователь создаёт `Run` из `Test Scenario`.
2. Система создаёт снимок сценария и разворачивает шаги в `Step Execution`.
3. Исполнитель последовательно отмечает ход выполнения.
4. По завершении шага фиксируются комментарии, артефакты и SAP-результаты.

### 9.5 Перепрогон шагов

1. Если шаг завершился неуспешно, исполнитель может запустить повторное выполнение.
2. Система создаёт новый `Step Execution` для этого шага (предыдущий сохраняется в истории).
3. Статус шага определяется последней попыткой.

### 9.6 Валидация

1. Для завершённого шага создаются или активируются валидации.
2. Валидатор отмечает результат каждой проверки.
3. При негативном результате может создаваться `Defect`.

### 9.7 Завершение прогона

`Run` считается завершённым, если:

- все обязательные шаги завершены терминальным статусом;
- все обязательные валидации завершены;
- по дефектам принято решение, достаточное для фиксации результата прогона.

## 10. Функциональные Требования

### FR-1. Управление сценариями

Система должна позволять:

- создавать, редактировать и архивировать сценарии;
- задавать последовательность шагов;
- копировать сценарий как основу для нового с сохранением `parent_id`;
- хранить версию сценария;
- редактировать сценарий в JSON-представлении с валидацией схемы.

### FR-2. Управление тестовыми планами

Система должна позволять:

- создавать план под проектную фазу;
- включать в план набор сценариев (many-to-many);
- отслеживать состояние каждого сценария внутри плана;
- видеть сводный прогресс по плану;
- рассчитывать прогресс плана по исполненным сценариям, а не по шагам;
- редактировать план в JSON-представлении с валидацией схемы.

### FR-3. Исполнение прогонов

Система должна позволять:

- запускать `Run` из `Test Scenario`;
- автоматически создавать снимок сценария при старте прогона;
- видеть последовательность шагов;
- отмечать статус каждого шага;
- фиксировать комментарий, время, артефакты и SAP-результаты шага;
- повторно выполнять отдельные шаги внутри текущего `Run` с сохранением истории попыток;
- блокировать или завершать прогон.

### FR-4. Валидации

Система должна позволять:

- задавать несколько валидаций на шаг;
- назначать валидатора;
- фиксировать статус и комментарий по каждой проверке;
- отделять факт исполнения от факта подтверждения результата.

### FR-5. Дефекты

Система должна позволять:

- создавать дефект из шага или валидации;
- хранить связь дефекта с исходным контекстом;
- отслеживать статус исправления и ретеста;
- быстро находить все дефекты по плану, сценарию, прогону или шагу.

### FR-6. Просмотр и отчётность

Система должна позволять:

- видеть список планов, сценариев, прогонов и дефектов;
- фильтровать по статусам, тегам и ответственным;
- смотреть dashboard по плану: всего сценариев / исполнено / failed / blocked, open / closed дефектов, прогресс в процентах;
- открывать карточку любого объекта с полной трассировкой связей.

### FR-7. История изменений

Для каждой ключевой сущности хранить:

- `created_at`
- `created_by`
- `updated_at`
- `updated_by`

Полный аудит изменений не обязателен для MVP, но структура данных не должна мешать его добавить позже.

### FR-8. Импорт и экспорт

Для MVP достаточно:

- экспорта данных в JSON;
- резервного копирования файловой директории;
- возможности ручного импорта подготовленных JSON-файлов;
- экспорта артефактов и JSON-снимков вместе с данными прогона.

### FR-9. Аутентификация

- Все рабочие экраны требуют аутентификации.
- MVP: индивидуальные локальные учётные записи с входом по `email + password`.
- Пароли хранятся только в виде `password_hash`.
- Post-MVP: разграничение прав по ролям и более детальная политика доступа.

## 11. Требования К Интерфейсу

UI должен быть:

- минималистичным;
- близким по восприятию к Jira;
- ориентированным на плотную работу со списками и карточками;
- удобным прежде всего на desktop;
- на английском языке.

Обязательные экраны MVP:

- список и карточка `Test Scenario`;
- список и карточка `Test Plan`;
- JSON-редактор для `Test Scenario` и `Test Plan`;
- экран исполнения `Run`;
- список и карточка `Defect`;
- dashboard по плану.

На экране `Run` должны быть видны:

- текущий шаг;
- статус каждого шага;
- комментарии;
- история попыток по шагу;
- валидации;
- зафиксированные SAP-объекты;
- связанные дефекты.

Dashboard по плану:

- всего сценариев / исполнено / failed / blocked;
- open / closed дефектов;
- прогресс в процентах, где `исполнено = количество сценариев с Run в статусе completed или blocked`, а `progress = исполнено / всего сценариев * 100`.

## 12. Требования К Хранению Данных

Основное ограничение: данных ожидается немного, поэтому MVP может использовать файловое хранение.

Требования:

- формат хранения: JSON;
- каждая сущность хранится независимо и имеет стабильный числовой `id` (автоинкремент);
- структура должна быть пригодна для резервного копирования и ручной диагностики;
- данные должны переживать перезапуск контейнера;
- вложения и артефакты хранятся в файловой системе с ссылками из JSON;
- пользовательские расширения схемы хранятся только в `custom_fields`.

Пример логической структуры:

```text
/data/scenarios/*.json
/data/plans/*.json
/data/runs/*.json
/data/defects/*.json
/data/users/*.json
/data/attachments/<run-id>/<step-id>/*
```

## 13. Нефункциональные Требования

- Развертывание на VPS `94.23.107.11`.
- Развертывание через Docker.
- Система должна быть понятной без длительного обучения.
- Основной сценарий работы должен занимать минимальное число кликов.
- Ошибки и блокировки должны быть видны сразу на уровне списка и карточки.
- Первый этап эксплуатации проходит на одном реальном проекте.
- Масштабирование между проектами — через отдельные изолированные Docker-развёртывания (один проект = один Docker stack).

## 14. Критерии Готовности MVP

MVP можно считать готовым, если система позволяет:

1. создать сценарий минимум из нескольких шагов;
2. скопировать сценарий и подставить конкретные параметры;
3. включить сценарий в тестовый план;
4. запустить прогон и пройти шаги с фиксацией статусов;
5. повторно выполнить отдельный шаг с сохранением истории;
6. выполнить хотя бы одну валидацию по шагу;
7. создать дефект из негативного результата;
8. увидеть итоговый статус прогона и прогресс плана после перезапуска системы;
9. загрузить вложение (скриншот или JSON) к шагу;
10. войти в систему под индивидуальной учётной записью по `email + password`.

## 15. Consultant Mode и SAP-интеграция (post-MVP)

Не входит в MVP, реализуется как первая итерация после базового ядра.

### 15.1 Назначение

Electron-приложение, которое запускает тот же веб-интерфейс, но дополнительно умеет подключаться к SAP-системе и запрашивать данные — для сбора доказательств выполнения тестов.

### 15.2 Основной сценарий

1. Консультант выполняет шаг теста в SAP.
2. В ProofForge по номеру документа запрашивает его содержимое из SAP.
3. Система скачивает документ в виде JSON и сохраняет как артефакт `Step Execution`.
4. Артефакт доступен в протоколе теста для просмотра, копирования и формального подписания результатов.

### 15.3 SAP-коннектор

Коннектор абстрагирован от протокола. Поддерживаемые транспорты:

- RFC/BAPI (через node-rfc или PyRFC);
- OData / HTTP;
- другие доступные механизмы.

Выбор транспорта зависит от того, что доступно в конкретной SAP-системе.

### 15.4 Поддерживаемые типы данных

Набор шаблонов запросов расширяется постепенно. Первый поддерживаемый тип — FI-документ.

### 15.5 Подписание результатов

Формальное подписание в MVP consultant mode — простая фиксация: пользователь нажимает "Approved", система сохраняет timestamp + username. Криптографическая подпись не требуется.

## 16. Зафиксированные Решения

1. `Test Case` как отдельная сущность убран из MVP. Вместо этого используется копирование `Test Scenario` с `parent_id` для трассировки.
2. Все сценарии равноправны. Шаблонность — вопрос конвенции (именование, теги), а не типа сущности.
3. При создании `Run` делается снимок сценария, чтобы изменения оригинала не влияли на запущенный прогон.
4. Перепрогон шага создаёт новый `Step Execution`, история попыток сохраняется.
5. Вложения в MVP обязательны: скриншоты и JSON-файлы загружаются вручную.
6. Аутентификация обязательна (публичный IP). MVP — индивидуальные учётные записи с входом по `email + password`.
7. Роли — теги без ограничений в MVP.
8. ID — числовой автоинкремент. Префиксы бизнес-процессов (OTC, R2R, P2P, E2E) — теги.
9. Связь Scenario ↔ Plan — many-to-many.
10. Статусы без жёстких ограничений на переходы в MVP.
11. Electron + SAP-коннектор — post-MVP, первая итерация после ядра.
12. SAP-коннектор абстрактный: RFC, OData, HTTP — что доступно.
13. Шаблоны SAP-запросов начинаются с FI-документа.
14. Дорожная карта интеграций: SAP → почта → Jira.
